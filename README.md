# Программный комплекс ассистивных веб-технологий «ClearVision»

## 📋 Общие сведения о проекте
**ClearVision** — это высоконагруженный асинхронный B2B-микросервис и кросс-доменная JavaScript-инъекция, предназначенная для динамической адаптации и трансформации цифрового контента под нужды инвалидов по зрению и лиц с когнитивными нарушениями.

Программный комплекс спроектирован и реализован в строгом соответствии с требованиями национального стандарта **ГОСТ Р 52872-2019** («Интернет-ресурсы и требования доступности для инвалидов по зрению»).

## 📐 Системная архитектура (DDD / Ports and Adapters)

В основу архитектурного каркаса заложены принципы **Чистой Архитектуры (Clean Architecture)** Роберта Мартина и методология **Гексагонального проектирования (Ports and Adapters)**. Это обеспечивает полную независимость бизнес-логики от внешних инфраструктурных фреймворков и баз данных.

```text
                          АРХИТЕКТУРНЫЕ СЛОИ СИСТЕМЫ
┌─────────────────────────────────────────────────────────────────────────────┐
│ 1. ENTRYPOINTS (FastAPI, Pydantic DTO, CORS Middleware)                     │
│    └─► Принимает внешние HTTP-запросы, валидирует контракты                 │
├─────────────────────────────────────────────────────────────────────────────┤
│    │ 2. SERVICE LAYER (UserProfileService, Use Cases, Unit of Work)         │
│    │    └─► Оркестрирует сценарии, координирует транзакции и UPSERT         │
├─────────────────────────────────────────────────────────────────────────────┤
│    │    │ 3. DOMAIN LAYER (UserProfile Entity, Валидация по ГОСТ)           │
│    │    │    └─► Чистая бизнес-логика в памяти, инварианты системы          │
├─────────────────────────────────────────────────────────────────────────────┤
│    │ 4. ADAPTERS LAYER (SQLAlchemy 2.0, PostgreSQL Repository)              │
│    │    └─► Паттерн Data Mapper, изоляция хранения данных                   │
└─────────────────────────────────────────────────────────────────────────────┘
```


### Подробное описание слоев:
1. **Domain Layer (Доменное ядро)**: Содержит доменную модель `UserProfile`. Инкапсулирует инварианты системы и ограничения ГОСТа. Реализована строгая многоуровневая валидация: технический предел масштабирования шрифта жестко зафиксирован на значении `3.0` для предотвращения разрушения DOM-дерева верстки.
2. **Service Layer (Сервисный слой)**: Выступает в роли диспетчера вариантов использования (Use Cases). Реализует логику атомарных операций **UPSERT** (идемпотентная вставка или обновление данных). Если профиль пользователя уже существует, слой выполняет обновление через контекст сессии (`session.merge`), исключая возникновение ошибок дублирования первичных ключей.
3. **Adapters Layer (Слой адаптеров)**: Реализует архитектурный паттерн *Repository*. Компонент `SqlAlchemyRepository` выполняет функцию *Data Mapper*, бесшовно преобразуя плоские строки реляционных таблиц базы данных в чистые доменные объекты Python и наоборот. 
4. **Entrypoints (Точки входа)**: Веб-слой на базе асинхронного веб-фреймворка **FastAPI**. Валидация входящих JSON-пакетов делегирована схемам **Pydantic v2** (`ProfileCreateSchema`). Слой оснащен глобальным конфигурационным менеджером `pydantic-settings` для безопасного считывания системных секретов из изолированных файлов среды `.env`.

## 🗄️ Проектирование Базы Данных (PostgreSQL)

В качестве основного хранилища данных выбрана объектно-реляционная СУБД **PostgreSQL 15**, развернутая в изолированном Docker-контейнере. 

### Спецификация таблицы `user_profiles`:
*   `user_id` (тип: `VARCHAR/String`, **PRIMARY KEY**) — уникальный анонимный токен пользователя, генерируемый на стороне фронтенда.
*   `font_size` (тип: `FLOAT/Numeric`) — коэффициент масштабирования шрифта (интервал от 1.0 до 2.0 в интерфейсе).
*   `visual_mode` (тип: `VARCHAR/String`) — строковый идентификатор цветовой схемы отображения (`normal` / `monochrome`).
*   `letter_spacing` (тип: `FLOAT/Numeric`) — величина межбуквенного интервала в пикселях.

**Политика живучести данных (Data Persistence):** Данные СУБД монтируются на физический диск хост-машины через механизм Docker Volumes (`postgres_data`). Настройки пользователей сохраняются перманентно и не сбрасываются при перезапуске контейнеров или отключении питания сервера.

## 🐳 Развертывание микросервиса (Docker Compose)

Проект полностью докеризирован и готов к многоконтейнерному продакшн-деплою. Инициализация сетевых мостов, баз данных и веб-сервера uvicorn автоматизирована.

Для сборки и запуска всей инфраструктуры выполните в корне проекта команду:
```bash
docker compose up --build
```

### Доступные сетевые интерфейсы:
*   `http://localhost:8080` — Внешняя точка входа микросервиса.
*   `http://localhost:8080/docs` — Интерактивная документация API (Swagger UI).
*   `http://localhost:8080/static/widget.js` — Путь автоматической раздачи фронтенд-компонента (статика FastAPI).
*   `Порт 5435` — Проброшенный наружу порт СУБД PostgreSQL 15.

## 🎨 Сценарий комплексной демонстрации (Фронтенд-инъекция)

Фронтенд-компонент представляет собой легковесный, оптимизированный скрипт на чистом JavaScript (Vanilla JS), работающий через **Shadow DOM**. Это полностью изолирует стили и элементы управления виджета от глобального CSS чужого сайта.

Программный комплекс обеспечивает бесшовный сквозной перенос настроек доступности конкретного человека при переходе между совершенно разными сайтами в сети.

### Шаг 1. Внедрение на первом ресурсе
1. Откройте любой сторонний веб-сайт (например, [example.com](http://example.com)).
2. Откройте консоль разработчика (**F12** -> вкладка *Console*).
3. Для снятия встроенной защиты браузера от вставки стороннего кода введите фразу `разрешить вставку` и нажмите *Enter*.
4. Выполните инъекцию нашего виджета, загрузив его по сети из работающего Docker:
   ```javascript
   const script = document.createElement('script');
   script.src = 'http://localhost:8080/static/widget.js';
   document.head.appendChild(script);
   ```
5. В углу экрана появится панель управления **ClearVision**.

### Шаг 2. Интерактивная настройка и проверка СУБД
1. Измените параметры панели: переключите размер шрифта на *Большой*, интервал букв на *Средний*, а цветовой режим — в *Черно-белый*. Сайт мгновенно трансформируется с помощью встроенных CSS-фильтров.
2. В этот момент виджет отправляет асинхронный `POST`-запрос на FastAPI. Логи контейнера зафиксируют успешный ответ со статусом **`201 Created`** — настройки записаны в базу данных Postgres.
3. Извлеките сгенерированный токен пользователя из памяти текущего сайта командой:
   ```javascript
   localStorage.getItem('clearvision_user_id');
   ```
   *(Скопируйте полученную строку, например: `"user_abc123xyz"`)*.

### Шаг 3. Симуляция сквозного перехода на второй сайт
*Браузеры изолируют хранилища localStorage разных доменов в целях безопасности. Для симуляции перехода пользователя на другую независимую интернет-площадку, подключенную к нашей экосистеме, выполните следующее:*

1. Откройте в новой вкладке совершенно другой сайт (например, Swagger UI нашего приложения: `http://localhost:8080/docs`).
2. Откройте консоль (**F12**).
3. Передайте токен сессии в память нового домена:
   ```javascript
   localStorage.setItem('clearvision_user_id', 'ВСТАВЬТЕ_СЮДА_ТОКЕН_ИЗ_ШАГА_2');
   ```
4. Запустите стандартный код инъекции виджета:
   ```javascript
   const script = document.createElement('script');
   script.src = 'http://localhost:8080/static/widget.js';
   document.head.appendChild(script);
   ```
5. **Результат**: Виджет сделает автоматический `GET`-запрос. СУБД PostgreSQL найдет профиль пользователя по первичному ключу, и второй сайт **мгновенно и автоматически перестроится** под те настройки, которые пользователь задал на первом сайте!

## 🧪 Статус автоматического тестирования
Система полностью спроектирована по методологии **TDD (Test-Driven Development)**. Запуск тестового покрытия: `pytest`.
*   `tests/unit/domain` — Верификация бизнес-правил ГОСТа.
*   `tests/unit/service_layer` — Тестирование Use Cases в изоляции через *Fake Репозиторий*.
*   `tests/unit/api` — Тестирование контрактов FastAPI через `httpx.AsyncClient` и `dependency_overrides`.
*   `tests/integration` — Тестирование интеграции с базой данных.
